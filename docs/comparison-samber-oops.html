Error Family
vs. Oops

go-error-family v0.3.0 — larsartmann

June 2026 · Comparative Analysis

Two Go libraries that ask the same question — "what do you do with an error?" — and arrive at radically different answers. One classifies behavior. The other enriches context. Here is how they differ, where each excels, and when to choose which.

01 — Philosophy

Different answers to the same question

go-error-family

Share the protocol,
not the implementation.
"What should the system do with this error?"

Behavioral classification first. Every error gets a Family — Rejection, Conflict, Transient, Corruption, or Infrastructure — which directly answers: should we retry? what exit code? whose fault? what tone for the user? The library provides small interfaces (Coded, Classified, Contextual, Retryable) that any error type can implement. It is a classification protocol, not a structured error wrapper.

samber/oops

Rich context for
every error.
"What do I know about this error?"

Observability and debugging first. Every error carries a rich payload — stack traces, source code fragments, structured attributes, trace/span IDs, user/tenant context, HTTP request/response bodies, timing data, ownership tags, and domain labels. It is a context enrichment layer that wraps any error into an OopsError with a fluent builder API. Designed to make production debugging dramatically easier.

02 — Error Model

How each library structures an error

go-error-family · Error struct 
type Error struct {
    code      string            // "db.timeout"
    message   string            // "query took too long"
    family    Family            // Transient
    context   map[string]string // {"host":"db1"}
    cause     error             // underlying error
    timestamp time.Time         // when created
}
samber/oops · OopsError struct 
type OopsError struct {
    err      error            // underlying error
    msg      string           // message
    code     any              // error code (any type)
    domain   string           // "database"
    tags     []string          // ["connection","timeout"]
    context  map[string]any   // {"query":"SELECT..."}
    trace    string           // ULID auto-generated
    span     string           // span ID
    hint     string           // debug hint
    public   string           // user-safe message
    owner    string           // responsible team
    userID   string           // user identifier
    tenantID string           // tenant identifier
    // + stacktrace, HTTP req/res,
    //   time, duration, source
    //   code fragments...
}

go-error-family asks "what kind of error is this?" and gives you a behavioral answer. samber/oops asks "what do I know about this error?" and gives you a forensic answer.

03 — Ergonomics

Creating and handling errors

go-error-family · CLI boundary 
// Create with explicit family
err := errorfamily.NewTransient(
    "db.timeout",
    "query took too long",
).WithContext("host", "db1.example.com")

// Classify any error — even stdlib
family := errorfamily.Classify(err)
// → Transient (retryable, exit 75)

// CLI boundary: one line in main()
func main() {
    if err := run(); err != nil {
        os.Exit(errorfamily.HandleError(err))
    }
}
// stderr: "A temporary error occurred..."
// exit: 75 (EX_TEMPFAIL)

// Register third-party sentinels
errorfamily.RegisterClassification(
    sql.ErrConnDone,
    errorfamily.Transient,
)
samber/oops · Builder pattern 
// Create with rich context
err := oops.In("database").
    Tags("connection", "timeout").
    Code("DB_CONN_TIMEOUT").
    With("host", "db1.example.com").
    Trace("req-abc-123").
    User("user-456").
    Errorf("connection failed after %dms", 5000)

// Access rich metadata
oopsErr, ok := oops.AsOops(err)
fmt.Println(oopsErr.Stacktrace())
fmt.Println(oopsErr.Sources())  // source code!
fmt.Println(oopsErr.Code())

// Logger integration (slog)
slog.Error("request failed",
    "error", err) // auto-structured

// Panic recovery
err = oops.Recover(func() {
    panic("something broke")
})

05 — Architecture

How they think about error handling

go-error-family — Protocol-first

Interface-driven

Four tiny interfaces (Coded, Classified, Contextual, Retryable) that any struct can implement. You never need to import the library's Error type — just satisfy the interface.

Classification cascade

Classify(err) walks: multi-error → Classified interface → Retryable interface → registered sentinels → default Transient. First match wins. Thread-safe sentinel registry with lock-free snapshots.

Boundary-oriented

Designed for the moment errors leave the program. HandleError() is the canonical entry point — classifies, formats with templates, writes to stderr, returns exit code. One line in main().

Diagnostic engine

Concurrent rule-based diagnostics (filesystem, network, git, postgres). Rules match on error codes/context, run targeted system checks, produce confidence-scored findings. Agent layer produces FixSteps without executing them.

Fail-open design

Unknown errors default to Transient (retryable). Classify(nil) returns Rejection. Every decision is deliberate about what happens at the boundary.

samber/oops — Enrichment-first

Builder-driven

Fluent API with 20+ methods. Each returns a new OopsErrorBuilder. Terminal methods (New(), Wrap(), Errorf()) materialize the error. Immutable builder — each call copies the context map.

Observability-native

Stack traces, source code fragments, trace/span IDs, OpenTelemetry integration, structured logging via slog.LogValuer. Every error is a rich telemetry event waiting to be emitted.

Layer chain

Wrapping creates layers accessible via Layers(). Context flattens across layers (later wins). Tags merge and deduplicate. Trace prefers explicit over auto-generated. "Deepest wins" for most scalar fields.

Multi-domain context

First-class support for HTTP req/res (with body capture), user/tenant identity (with attributes), ownership tagging, domain categorization, and timing data. Built for microservice and multi-tenant environments.

Copy cost tradeoff

The immutable builder copies the context map at each step — O(n) per builder call. Optimized for the common case: create error, rarely introspect. Benchmarks confirm allocation patterns scale with builder chain depth.

07 — Verdict

When to choose which

These libraries are not competitors. They solve different problems and can even be used together — wrap errors with oops for observability, then classify them with error-family for behavioral decisions.

Choose go-error-family when

Best for: CLI tools, services, API boundaries
  • Your program needs to decide what to do with errors (retry, exit, alert)
  • You need consistent exit codes across CLI tools
  • You want automatic classification of third-party errors
  • Your error model is about behavior (retryable? user's fault?)
  • You need diagnostic rules that investigate system state
  • You want zero-dependency, protocol-first design
  • You need user-facing messaging with tone/audience targeting
  • You want to share a protocol across services without sharing implementations

Choose samber/oops when

Best for: HTTP services, microservices, observability
  • You need rich debugging context in production (stack traces, source code)
  • Your errors flow through structured logging pipelines
  • You need distributed tracing with trace/span IDs
  • You want user/tenant context attached to errors
  • You need HTTP request/response capture for debugging
  • You want JSON serialization of error context
  • You value a fluent builder API for error construction
  • You need OpenTelemetry integration out of the box
Complementary use
Using both together 
// Enrich with oops for observability
richErr := oops.In("database").
    Trace("req-123").
    With("host", "db1").
    Wrap(dbErr)

// Classify with error-family for behavioral decisions
family := errorfamily.Classify(richErr)
if family.IsRetryable() {
    scheduleRetry(richErr)
}
08 — Tradeoffs

What you gain and what you give up

Tradeoff go-error-family samber/oops
Conceptual model Simple: 5 families, clear semantics. Easy to explain to a team. Rich: many dimensions. More to learn, more to configure.
Custom error types Implement 1–4 interfaces on your own struct. Full control. Wrap into OopsError. Your struct is hidden behind the wrapper.
Context value types Strings only — simple, serializable, no reflection. any — flexible but requires serialization thought.
Production debugging Diagnostic rules check live system state. No stack traces. Stack traces + source fragments. Rich forensic data.
Operational decisions First-class: retry, exit codes, audience, tone. The whole point. Not provided. You build retry/exit logic yourself.
Dependency footprint Zero. Only stdlib.vet 3 deps (ulid, lo, otel/trace). Larger binary impact.
Maturity / adoption Young (v0.3.0, 80 commits). Niche but focused. Mature (v1.22.0, 956 stars, 41 releases). Proven in production.
Go version gate 1.26+ only — errors.AsType requirement. 1.21+ — broader compatibility.
