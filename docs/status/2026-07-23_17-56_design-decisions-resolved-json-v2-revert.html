Performance

writeHTTPError double-classifies every HTTP error

http.go:66,82 — Classify(err) called at line 66, HTTPStatus(err) calls Classify(err) again internally

writeHTTPError calls family := Classify(err) for the response body, then calls w.WriteHeader(HTTPStatus(err)) which internally calls Classify(err) again. Every HTTP error response now runs the classification pipeline twice. The fix is to compute the status code once and reuse the family: 

// Current (double classify):
family := Classify(err)
// ...
w.WriteHeader(HTTPStatus(err)) // calls Classify(err) again

// Should be (single classify):
family := Classify(err)
status := family.HTTPStatus()
if hs, ok := errors.AsType[HTTPStatuser](err); ok && hs.HTTPStatus() != 0 {
    status = hs.HTTPStatus()
}
w.WriteHeader(status)
Impact: Minor (Classify is ~285ns/0-allocs with 50 sentinels), but it's a code smell. The whole point of the atomic.Pointer optimization was to make Classify cheap — and now we call it twice on the hot path.
