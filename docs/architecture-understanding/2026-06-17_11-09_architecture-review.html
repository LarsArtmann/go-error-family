go-error-family
Scalability & Modularity

Evaluates the module boundaries, dependency directions, coupling, and composability of github.com/larsartmann/go-error-family at v0.5.0 — after the pro/contra review execution. Includes D2 diagrams of the current and ideal architecture.

The bottom line

One god-module, four submodules, one clear path to v1.0

The classification core is excellent — zero-dep, well-tested, copy-on-write immutable. But it shares a go.mod with three experimental packages (agent/, diagnose/, examples/). This prevents the core from reaching v1.0 stability independently. Extracting the two experimental packages into their own modules is the single highest-leverage architectural change.

Strengths

What the architecture gets right

STR 01 Zero-dependency core — the hardest constraint, achieved high impact

The root go.mod has zero external dependencies. No samber/oops, no OpenTelemetry, no testify. This means any Go project can adopt the classification core without pulling transitive dependency trees. The bridge module (which pulls in oops) is opt-in. This is textbook "mechanism, not policy" — the core defines what classification means; consumers bring how they want their errors enriched.

go.mod (root) — zero require directives
STR 02 Clean DAG — all dependency arrows point toward the core high impact

Every internal dependency points toward the classification core. No cycles exist. The agent depends on diagnose (which depends on core). The bridge depends on core. The git/postgres submodules depend on core + diagnose. This is the correct direction — domain types define the interface, infrastructure implements it. 

agent/    → diagnose/ → core (root)
bridge/   → core (root)
git/      → core + diagnose
postgres/ → core + diagnose
examples/ → core (leaf nodes)
go.work · all go.mod files
STR 03 Interfaces are the product — composable with any error type high impact

Four small interfaces (Coded, Classified, Contextual, Retryable), each embedding error, define the protocol. Any error type implementing them works with Classify, HandleError, and ExitCode. The bridge module proves this — it makes samber/oops errors satisfy the interfaces without either library knowing about the other. This is Unix-pipe composability.

interfaces.go · bridge/bridge.go
STR 04 Copy-on-write errors — safe for concurrent use med impact

After v0.5.0, WithContext, WithCause, and WithTimestamp all return new *Error pointers. This eliminates the data race where shared/sentinel errors were mutated by chaining. Errors are now safe to share across goroutines.

error.go:120-152 (clone helper + copy-on-write methods)
STR 05 Injectable Registry — test isolation without global state med impact

The Registry type (new in v0.5.0) replaces global mutable maps with a construct-and-pass pattern. Tests get isolation (no t.Cleanup(Unregister...) needed), and services get scoped error handling. DefaultRegistry preserves backward compatibility.

registry.go · HandleConfig.Registry field
Weaknesses

What holds the architecture back

CON 01 God-module: stable core shares go.mod with experimental packages high impact

The root module contains four distinct concerns: (1) the stable classification core, (2) the experimental diagnostic engine (diagnose/), (3) the experimental agent (agent/), and (4) examples (examples/cmd/). This means the core cannot reach v1.0 independently — a breaking change to the agent forces a version bump for the entire module, including the stable core. The v1.0 dream requires extracting agent/ and diagnose/ into their own modules. 

root go.mod
├── errorfamily/     ← STABLE: should be v1.0
├── diagnose/        ← EXPERIMENTAL: should be own module
├── agent/           ← EXPERIMENTAL: should be own module
└── examples/cmd/    ← LEAF: stays in root, compiled by CI
go.mod (root) · go.work
CON 02 Stale submodule version pins — submodules reference v0.3.0, not v0.5.0 high impact

All three submodules (bridge, diagnose/git, diagnose/postgres) pin the root dependency at v0.3.0. But v0.5.0 introduced breaking changes: copy-on-write methods, Registry type, template syntax change, and Compose removal. The submodules work locally via go.work (which overrides version pins), but external consumers pulling these submodules from the Go proxy would get v0.3.0 root — missing the breaking changes.

bridge/go.mod · diagnose/git/go.mod · diagnose/postgres/go.mod — all pin v0.3.0
CON 03 agent → diagnose coupling within the same module med impact

The agent package imports diagnose.DiagnosticResult and diagnose.StatusFailed directly. Today this is invisible — both are in the same module. But if either is extracted to its own module, the agent must explicitly depend on diagnose. This is a correct dependency direction (agent consumes diagnostic results), but the coupling should be through a thin interface, not concrete types. 

agent/agent.go:26  import ".../diagnose"
agent/agent.go:73  diagnosis []*diagnose.DiagnosticResult
agent/agent.go:120 if d.Status == diagnose.StatusFailed
agent/agent.go · diagnose/diagnose.go
CON 04 No internal/ usage — all packages are public export low impact

Every package in the root module is importable by any external consumer. There is no internal/ directory. The diagnose/mock.go (test helper) is public. The applyContext function in handle.go is unexported (good), but package-level visibility means the package itself is public API. After module extraction, internal/ should be used for test helpers and implementation details.

diagnose/mock.go · diagnose/helpers.go
Path forward

Recommendations

Ordered by impact on the path to v1.0. The first two are the single highest-leverage architectural change.

  1. Extract agent/ and diagnose/ into separate modules. The root module shrinks to the classification core only (~1000 LOC). Agent depends on diagnose (explicit go.mod require). Diagnose depends on core. Each versions independently. This unblocks the v1.0 stable core without freezing the experimental packages.
  2. Bump submodule version pins to v0.5.0. All three submodules must pin root at v0.5.0 (not v0.3.0). Run go mod tidy in each submodule directory. Verify with GOWORK=off go build ./... in each.
  3. Define the agent's dependency on diagnose through a thin interface. Instead of importing []*diagnose.DiagnosticResult directly, define a DiagnosticResult interface or move the type to core. This allows the agent to consume results from any diagnostic engine, not just this library's.
  4. Use internal/ for test helpers after module extraction. diagnose/mock.go should be in diagnose/internal/mock.go or a separate diagnose/testutil module. Implementation details that are currently package-level public should move behind internal/.
  5. Update CI for per-module parallelism. After module extraction, CI should test each module in parallel. The current CI already does this for the submodules — add steps for the new agent/ and diagnose/ modules.
  6. Ship v1.0 of the classification core. Once the core module contains only Family, Error, interfaces, Classify, Registry, HandleError, and constructors — freeze it. Tag it v1.0.0. The experimental modules (agent, diagnose, bridge, git, postgres) stay at v0.x.
