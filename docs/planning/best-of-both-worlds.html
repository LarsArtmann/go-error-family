00
Foundations

Design Axioms

These constraints shape every decision below. Violating any axiom weakens the integration.

Axiom 01

Thin Bridge, Never a Fork

The integration layer is an adapter — a few dozen lines that connect interfaces. Never embed, never fork, never vendor. Both libraries update independently.

Axiom 02

Own Different Things

Oops owns context (what happened, where, who, when). Error-family owns behavior (retry? exit? whose fault?). No overlap, no ambiguity about who owns what.

Axiom 03

Interface, Not Inheritance

Go composition over inheritance. The bridge uses interfaces both libraries already expose. No shared base type, no type coupling. Either library can be swapped without the other noticing.

Axiom 04

Both Optional, Either First

Use error-family alone. Use oops alone. Use both. The integration is additive — it never requires both to function, and it never degrades either when used solo.

01
Phase 1 · Immediate

Bridge Adapter

L1
Ship Today · ~30 lines

ClassifiedOops — A Thin Wrapper

Make OopsError satisfy error-family's Classified and Retryable interfaces.

The simplest possible integration: a wrapper struct that carries an OopsError and a Family. It delegates Error(), Unwrap(), and ErrorContext() to oops while exposing ErrorFamily() and IsRetryable() for error-family's classification cascade.

bridge.go 
package bridge

import (
    "github.com/larsartmann/go-error-family"
    "github.com/samber/oops"
)

// ClassifiedOops wraps an OopsError with a behavioral Family.
// Implements Classified + Retryable + Contextual for error-family's Classify().
type ClassifiedOops struct {
    *oops.OopsError
    family errorfamily.Family
}

func Wrap(err *oops.OopsError, f errorfamily.Family) *ClassifiedOops {
    return &ClassifiedOops{OopsError: err, family: f}
}

func (c *ClassifiedOops) ErrorFamily() errorfamily.Family { return c.family }
func (c *ClassifiedOops) IsRetryable() bool               { return c.family.IsRetryable() }

// ErrorContext bridges oops.Context() → map[string]string for the Contextual interface.
func (c *ClassifiedOops) ErrorContext() map[string]string {
    raw := c.Context()
    out := make(map[string]string, len(raw))
    for k, v := range raw {
        if s, ok := v.(string); ok {
            out[k] = s
        }
    }
    return out
}
Note

Classify() checks the Classified interface before the Retryable interface in its cascade (step 2 vs step 3). This means ClassifiedOops is picked up immediately — no sentinel registration needed, no ambiguity.

usage.go — CLI boundary with both libraries 
// Enrich with oops
rich := oops.In("database").
    Tags("connection", "timeout").
    With("host", "db1.example.com").
    Trace("req-abc-123").
    Wrap(dbErr)

// Attach family
classified := bridge.Wrap(oopsErr, errorfamily.Transient)

// Now both systems work:
errorfamily.IsRetryable(classified) // → true (from Family)
errorfamily.ExitCode(classified)   // → 75 (EX_TEMPFAIL)
classified.Stacktrace()              // → full oops stacktrace
classified.Trace()                    // → "req-abc-123"

Phase 2 · Next Iteration

Context-Driven Classification

L2
Next Iteration · ~80 lines

Derive Family from Oops Metadata

Replace manual family assignment with inference from oops' domain, tags, and context.

Instead of hard-coding errorfamily.Transient at every call site, build a classification function that reads oops' structured metadata — domain, tags, error codes, hints — and derives the family. This makes the mapping centralized, testable, and evolveable without touching call sites.

classify.go — Infer family from oops metadata 
package bridge

func InferFamily(err *oops.OopsError) errorfamily.Family {
    // 1. Explicit tag wins
    for _, tag := range err.Tags() {
        switch tag {
        case "retryable":  return errorfamily.Transient
        case "conflict":   return errorfamily.Conflict
        case "corruption": return errorfamily.Corruption
        case "rejection":  return errorfamily.Rejection
        }
    }

    // 2. Domain-based defaults
    switch err.Domain() {
    case "validation", "auth":
        return errorfamily.Rejection
    case "database", "network", "cache":
        return errorfamily.Transient
    case "storage":
        return errorfamily.Infrastructure
    }

    // 3. Fail-open: unknown → retryable
    return errorfamily.Transient
}

// AutoWrap infers family and creates a ClassifiedOops in one step.
func AutoWrap(err *oops.OopsError) *ClassifiedOops {
    return Wrap(err, InferFamily(err))
}
Note

The inference cascade mirrors error-family's own: explicit → structural → default. Tags are the most intentional signal (developer chose them), domain is structural, and the default is Transient — consistent with error-family's fail-open philosophy.

Mapping Table

Domain → Family Defaults

  • validation, auth → Rejection
  • database, network, cache → Transient
  • storage, infra → Infrastructure
  • data, schema → Corruption
  • anything else → Transient (fail-open)
Override Mechanism

Tag-Based Override

  • Tags are developer-intentional — checked first
  • Any oops tag matching a family name overrides the domain default
  • .Tags("retryable") forces Transient regardless of domain
  • Zero-cost when unused — no tags means pure domain inference

Phase 3 · Ecosystem

Unified Boundary Handler

L3
Ecosystem Vision · ~200 lines

The Handle Package — Orchestrate Both

A single entry point that enriches, classifies, diagnoses, analyzes, and emits.

The architectural sweet spot. A handle package that sits at the program boundary — main(), HTTP middleware, gRPC interceptor — and orchestrates all five steps in sequence. Each step is optional. Each step uses the best tool for its job. The result: one function call that produces a complete error response with behavioral decisions, forensic context, diagnostic findings, and structured logging.

handle.go — The unified handler 
package handle

type Config struct {
    Enrich    bool            // attach oops context (stack, trace, user)
    Classify  bool            // derive family + exit code
    Diagnose  bool            // run diagnostic rules
    Analyze   bool            // run debug agent for root cause
    Logger    *slog.Logger     // structured log destination
    Output    io.Writer       // user-facing message output (stderr)
    Rules     []diagnose.DiagnosticRule
}

type Result struct {
    ExitCode    int
    Family      errorfamily.Family
    Message     string           // user-facing (What/Why/Fix/WayOut)
    FixSteps    []agent.FixStep
    Findings    []diagnose.DiagnosticResult
    Enriched    *oops.OopsError   // full forensic context
}

func Error(ctx context.Context, err error, cfg Config) *Result {
    r := &Result{}

    // 1. Enrich — attach forensic context
    if cfg.Enrich {
        r.Enriched = oops.In(domain(err)).
            TraceFromContext(ctx).
            Wrap(err)
    }

    // 2. Classify — behavioral decision
    if cfg.Classify {
        classified := bridge.AutoWrap(r.Enriched)
        r.Family   = errorfamily.Classify(classified)
        r.ExitCode = r.Family.ExitCode()
    }

    // 3. Diagnose — check live system state
    if cfg.Diagnose && len(cfg.Rules) > 0 {
        runner := diagnose.NewRunner(cfg.Rules...)
        r.Findings, _ = runner.Run(ctx, err)
    }

    // 4. Analyze — root cause + fix suggestions
    if cfg.Analyze && len(r.Findings) > 0 {
        ag := agent.New(agent.Config{Enabled: true})
        result, _ := ag.Analyze(ctx, err, r.Findings)
        r.FixSteps = result.FixSteps
    }

    // 5. Emit — structured log + user message
    if cfg.Logger != nil && r.Enriched != nil {
        cfg.Logger.ErrorContext(ctx, "request failed",
            "error", r.Enriched,
            "family", r.Family,
            "exit_code", r.ExitCode)
    }

    return r
}
main.go — One-line boundary 
func main() {
    if err := run(); err != nil {
        result := handle.Error(ctx, err, handle.Config{
            Enrich:   true,
            Classify: true,
            Diagnose: true,
            Analyze:  true,
            Logger:   logger,
            Output:   os.Stderr,
            Rules:    myRules(),
        })

        fmt.Fprintln(os.Stderr, result.Message)
        os.Exit(result.ExitCode)
    }
}
04
Guard Rails

What Not to Do

Anti-Pattern Why It's Wrong Instead
Don't Fork oops into error-family Different authors, release cadences, and audiences. Forking creates a maintenance monster that drifts from both upstreams. Keep them separate. Connect via a thin bridge package.
Don't Build a unified error type A mega-struct with all fields from both libraries is the worst of both worlds: bloated, opinionated, and impossible to evolve. Use composition. OopsError carries context, Error carries family. Bridge via interface.
Don't Couple at the type level If error-family imports oops (or vice versa), you can't use one without the other. Zero-optional becomes zero-mandatory. The bridge package is the only place that imports both. Everything else imports one or the other.
Don't Duplicate context in both Setting WithContext("host", "db1") on both oops and error-family is redundant and creates sync bugs. Oops owns context. Error-family reads it via the bridge. Single source of truth.
Don't Use oops for retry decisions Oops has no classification model. Tag-checking if hasTag("retryable") is ad-hoc and doesn't compose with third-party errors. Use errorfamily.Classify() — it handles multi-error, sentinels, interfaces, and defaults.
Consider Bridge as separate module If the bridge gains weight, it becomes a coupling point. But for ~30–200 lines, a sub-package is fine. Start as bridge/ in your app. Extract to a library only if multiple services need it.

Boundaries

Who Owns What

samber/oops owns

Context & Observability

  • Stack traces and source code fragments
  • Trace ID, span ID, correlation IDs
  • User and tenant identity
  • HTTP request/response capture
  • Structured attribute storage
  • Logger integration (slog, zap, logrus)
  • JSON serialization
  • Panic recovery
  • Domain tags and ownership
  • Timestamps and durations
go-error-family owns

Behavior & Decisions

  • Error family classification
  • Retry decision (IsRetryable)
  • Exit code mapping (BSD sysexits.h)
  • CLI boundary handler (HandleError)
  • User-facing messaging (What/Why/Fix/WayOut)
  • Audience targeting (User/Ops/All)
  • Tone selection (instructional, reassuring, urgent...)
  • Third-party sentinel registration
  • Diagnostic rules (filesystem, network, git, postgres)
  • AI debug agent and FixSteps
Bridge

The bridge package owns one thing: mapping between the two domains. InferFamily() converts oops metadata → error-family classification. ClassifiedOops satisfies error-family interfaces using oops data. That's the entire contract. No more, no less.
