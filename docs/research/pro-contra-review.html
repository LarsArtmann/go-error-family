The bottom line

A genuinely novel, high-craftsmanship library that solves a real problem nobody else solves — but its bleeding-edge Go requirement and pre-1.0 API churn are real adoption headwinds.

The core idea — a behavioral Family that answers “should I retry?”, “what exit code?”, and “whose fault is it?” in one concept — is the differentiator. No major Go error library provides this. The engineering hygiene (zero-dependency core, 96% coverage, fuzz tests, strict lint, exemplary documentation) is excellent. The main risks are a hard dependency on Go 1.26’s errors.AsType[T] (limits the addressable audience), global mutable registries, and a scope (classification + diagnostics + “agent”) that is ambitious for a v0.x library. Ship a stable v1.0 of the classification core first.

Strengths

What it gets right

The library’s value lives in one insight, executed with unusually disciplined engineering. The code, tests, and documentation are all above the bar for an open-source Go library.

PRO 01 A genuinely novel niche: behavioral classification as a first-class concept high impact

Every Go program hits if err != nil — but then what? pkg/errors adds stack traces. cockroachdb/errors makes errors network-portable. samber/oops adds structured context. emperror routes errors to handlers. None of them answer the three questions a CLI/HTTP boundary actually needs: should I retry? what exit code? whose fault is it? go-error-family’s Family answers all three in one concept. This is a real, differentiated value proposition, not a re-skin of existing work.

PRO 02 “Share the protocol, not the implementation” — the interfaces are the product high impact

Four small consumer interfaces — Coded, Classified, Contextual, Retryable — each embedding error. You don’t have to use the built-in Error struct; implement the interfaces you need. This is idiomatic Go (accept interfaces, return structs) and makes the library play well with arbitrary existing error types. The built-in Error is a convenience, not a tax.

interfaces.go · error.go · README “Implementing Your Own Error Type”
PRO 03 Zero-dependency core with surgical module boundaries high impact

The root module (go.mod) has zero external dependencies — it uses only the standard library. The heavier integrations are isolated in opt-in submodules: bridge (pulls in samber/oops), diagnose/git and diagnose/postgres (each their own module). go.work ties them together for local development. Submodules pin the root at a valid lower bound (v0.3.0) so MVS resolves correctly. This is sophisticated, mature multi-module hygiene for a Go library.

PRO 04 Engineering honesty: the “analysis-only” agent high impact

The DebugAgent interface has a single Analyze method that proposes fixes but does not execute them. An earlier ApplyFixes method that ran shell commands was identified as fraud (“a library should never run mkdir/git on behalf of callers”) and removed. This is the kind of scope discipline most libraries lack. The consumer decides what to do with FixStep.Command.

PRO 07 Principled, predictable classification cascade med impact

Classify(err) checks five sources in order, first match wins: multi-error (errors.Join) → Classified interface → Retryable interface → registered sentinels → default. The multi-error behavior is fail-closed: if any sub-error is non-retryable, the whole operation is non-retryable. The fail-open default (unknown → Transient) is a defensible opinion, documented and intentional.

PRO 08 The bridge module proves the protocol is library-agnostic med impact

bridge.Wrap(err, family) attaches a Family to a samber/oops error, preserving its stack trace and context, while satisfying the Classified/Coded/Retryable/Contextual interfaces — without either library knowing about the other. AutoWrap even infers the Family from oops tags/domains. This validates the core claim: the protocol composes with any error enrichment library you already use.

Weaknesses

What holds it back

These are not careless bugs — most are documented and intentional. But they are real frictions for adopters, and several carry real risk. They are ranked by impact on the library’s stated goal of broad adoption.

CON 01 Hard dependency on Go 1.26 limits the addressable audience high impact

The library requires Go 1.26 for errors.AsType[T](). As of release, Go 1.26 is bleeding-edge — most production codebases run on the previous stable release. A library at v0.x that demands the absolute latest Go version creates a real adoption barrier: teams on older toolchains cannot use it, and the feature (errors.AsType) is used pervasively, so there is no easy fallback path. For a library seeking uptake, this is the single biggest headwind. It is a deliberate bet on the modern idiom — but the bet narrows who can play. 

// classify.go — pervasively used, no fallback
if c, ok := errors.AsType[Classified](err); ok { ... }
if r, ok := errors.AsType[Retryable](err); ok { ... }
go.mod (go 1.26.3) · classify.go · handle.go · diagnose/helpers.go
CON 02 Global mutable registries are a library smell high impact

Two package-level mutable maps — registry.entries (classification) and templateRegistry.entries (message templates) — are mutated via Register* functions, typically from init(). They are mutex-protected and the read path snapshots before iterating (good), but global mutable state in a library is still a smell: it creates hidden coupling between packages, it cannot be scoped per-test without explicit Unregister cleanup, and parallel tests can interfere. The test code already bears the symptom — t.Cleanup(func() { UnregisterClassification(sentinel) }) appears throughout the suite. A registry-bearing type you construct and pass in would be cleaner and testable by construction.

classify.go:119-139 · handle.go:328-340 · classify_test.go:46
CON 03 WithContext mutates in place — not immutable high impact

The chaining API err.WithContext("k", "v").WithContext("k2", "v2") looks immutable, but WithContext returns the same pointer after mutating the internal map. If an error is shared (e.g. a package-level sentinel, or an error stored in a struct field), adding context mutates it for everyone — a data race under concurrency and a source of surprising aliasing. For a library that preaches immutability, this is a contradiction. The 0-alloc benchmark result is itself the tell: no copy means no isolation. 

func (e *Error) WithContext(key, value string) *Error {
    if e.context == nil { e.context = make(map[string]string) }
    e.context[key] = value  // mutates receiver in place
    return e                // returns same pointer
}

CON 04 The “agent” is deterministic, not an AI agent med impact

The package is named agent, the type is DebugAgent, and the README calls it an “AI debug agent”. But deterministicAnalyze is a fixed rule-synthesizer: it walks the diagnostic results, copies StatusFailed summaries, and parses shell commands out of prose. There is no probabilistic model, no learning, no inference — it is deterministic. The naming oversells the capability and may mislead adopters expecting pluggable LLM-backed analysis. A name like RCA (root cause analysis) or Synthesizer would be honest.

agent/agent.go:103-146 · README “AI debug agent”
CON 05 extractCommand parses prose with fragile heuristics med impact

The agent’s extractCommand reverse-engineers a shell command from a human-readable diagnostic suggestion by matching $ prefixes, Run: prefixes, indented lines, and a hardcoded list of command prefixes (git, mkdir, nc…). This is stringly-typed parsing of free-form prose. It will silently break the moment a diagnostic rule changes its suggestion format, and the looksLikeCommand heuristic will misclassify any new tool not in the list. The fix is structural: diagnostic rules should emit a structured {summary, command, rationale} triple, not prose to be parsed back.

agent/agent.go:148-197
CON 06 Retry is a single boolean — coarse for real-world semantics med impact

Only Transient is retryable; everything else is not. This is clean, but real retry logic is nuanced: a DB deadlock is retryable a few times; a connection refused needs backoff; a write conflict is retryable only if idempotent; a rate-limit is retryable after a delay. The library gives callers a binary signal and leaves all the hard parts (limits, jitter, idempotency, circuit-breaking) to be re-implemented everywhere. The Retryable interface returns bool, not a policy. That’s fine for a classification primitive, but the README’s “schedule a retry with backoff” framing implies more than the API delivers.

CON 07 {{.key}} looks like text/template but isn’t med impact

MessageTemplate uses {{.key}} placeholders resolved by naive strings.ReplaceAll — no escaping, no pipelines, no conditionals. The syntax is the same as Go’s text/template, so users will assume it behaves like it (and then hit no HTML escaping, which the AGENTS.md itself flags as an XSS concern). Either use real text/template (with template.HTML escaping opt-in), or pick a syntax that does not collide with the standard library. The current middle ground is the worst of both. 

// handle.go — looks like text/template, behaves like ReplaceAll
func applyContext(template string, context map[string]string) string {
    s := template
    for k, v := range context {
        s = strings.ReplaceAll(s, "{{."+k+"}}", v) // no escaping
    }
    return s
}

CON 08 Several intentional-but-surprising edge cases low impact

A cluster of opinionated choices, each defensible but unintuitive:

  • Classify(nil) → Rejection (“nil = caller’s fault”). ExitCode(nil) → 0. The asymmetry is intentional but trips up the unwary.
  • errors.Is matches on code + family only, ignoring message — two errors with the same code but different messages are “equal”.
  • Wrap(nil, …) → nil — you cannot construct an error wrapping a nil cause.
  • ParseFamily("garbage") → Transient — unrecognized input silently becomes retryable.

Credit: every one of these is documented in the “Surprising Behaviors” section. The critique is that a library with this many intentional surprises needs its adopters to read that section carefully — which not all will.

AGENTS.md “Surprising Behaviors” · classify.go · error.go:39-45
CON 09 Compose is acknowledged dead weight low impact

Compose(errs...) is literally return errors.Join(errs...) with a doc comment admitting it exists “for API discoverability”. Wrapping a stdlib function with a new name is API noise: it adds a thing to learn, document, and maintain, that does nothing the stdlib doesn’t. Honest move to document it; better move to remove it and point users at errors.Join in the README. 

func Compose(errs ...error) error {
    return errors.Join(errs...) // zero added logic
}

CON 10 Pre-1.0 with repeated breaking changes — adopters must accept churn med impact

At v0.4.0 the library has already shipped multiple breaking changes across four minor versions: rules modularized into submodules (v0.2.0), HandleConfig.Diagnose removed and agent.Config.Enabled return type changed (v0.4.0). This is expected and healthy for a v0 library actively finding its shape — but it means any adopter today is signing up for migration work on each release. The path to confidence is a deliberate v1.0 that freezes the core classification API.

Competitive position

Where it stands

go-error-family is not competing with context/wrapping libraries — it composes with them (the bridge module proves this). It occupies its own category: behavioral classification at the system boundary. No peer provides that out of the box.

Path forward

Recommendations

Ordered by impact on adoption and stability. The first three would materially change the library’s trajectory toward a credible v1.0.

  1. Ship a v1.0 of the classification core and freeze it. The root package (Family, Classify, ExitCode, IsRetryable, constructors, interfaces) is already excellent and stable in spirit. Cut v1.0 for just that, mark the diagnostics/agent/bridge as v0 experimental. A stable core is the single biggest unlock for adoption — it lets teams commit without fear of churn.
  2. Resolve the Go 1.26 tension deliberately. Either own it (document Go 1.26 as a firm requirement and lean into being modern), or provide a errors.As-based fallback build tag so the core runs on Go 1.23+. The current middle (requires latest, no fallback) is the highest-friction option.
  3. Make WithContext copy-on-write (or document clearly that Error is single-owner). A chaining API that looks immutable but mutates in place is a data-race waiting to happen for anyone who stores an error in a shared location. The 0-alloc benchmark is not worth the footgun.
  4. Replace the global registries with an injectable registry type. A *Classifier (or *Registry) you construct and pass in removes hidden global coupling, makes parallel tests trivially safe, and eliminates the t.Cleanup(Unregister…) tax across the test suite. Keep the package-level defaults for convenience.
  5. Rename the “agent” to match what it does. It is a deterministic root-cause synthesizer. RCA, Synthesizer, or DiagnosticAnalyzer would not oversell. If real pluggable AI analysis is the vision, make the interface accept an injectable analyzer so an LLM backend can be swapped in.
  6. Make diagnostic rules emit structured fix data, not prose. DiagnosticResult should carry a typed {summary, command, rationale} so the agent doesn’t have to reverse-parse shell commands out of human-readable strings with a hardcoded prefix list. This kills the fragile extractCommand heuristic at the root.
  7. Choose a template syntax and own it. Either adopt real text/template (with explicit escaping semantics) or rename the placeholders to something that doesn’t collide with the stdlib (e.g. {key}). The current {{.key}}-that-isn’t-template is the most confusing option.
  8. Drop Compose. Point users at errors.Join in the README. One less API surface to document, test, and explain.
